# SDD+TDD Workflow

**Core philosophy: SDD manages "what to build", TDD manages "did we build it right".**

```
     DESIGN (SDD)                          BUILD (TDD per task)
spec.md -> plan.md -> task.md    ->    RED -> GREEN -> REFACTOR
(what)     (how)      (steps)          (test)  (impl)   (clean)
[Owner]    [Owner/async]  [QA]         x N agents in parallel
```

---

## Governance Roles

| Role | Who | Core Responsibility |
|------|-----|---------------------|
| **Product Owner** | Human | Strategy, architecture decisions, key approvals |
| **Dispatcher** | Main Claude (Opus) | Intent parsing, routing, progress, executive summaries |
| **Architect** | Opus agent | Write spec/plan, tech selection, architecture design |
| **Engineer** | Alpha/Beta/Gamma (Sonnet) | TDD-driven implementation |
| **QA** | code-reviewer + security-reviewer + tdd-guide | Quality gates |
| **Scribe** | Haiku agent | Documentation, progress tracking |

### Owner Intervention Points

**Blocking (must approve before proceeding):**
- spec.md approval -- highest priority. Once approved, Agent Team proceeds autonomously
- Architectural plan.md -- new core components, dependency changes
- Pre-release final review -- before merging to main

**Non-blocking (async, does not block agents):**
- Non-architectural plan.md
- task.md -- QA handles
- Routine code review -- QA handles

```
|   +-- changelog.md     # Document change log

|   +-- integration/
+-- msg/                 # Custom message definitions (ROS2)
+-- srv/                 # Custom service definitions (ROS2)
+-- launch/              # Launch files

```

## Phase 1: Specification (`/sdd init <desc>`)

**Agent**: Architect (Opus)
**Output**: `.sdd/spec.md`

1. Research existing codebase -- understand current topology and interfaces
2. Brainstorm with Owner -- resolve ambiguity
3. Write spec -- acceptance criteria MUST be testable assertions
4. Prepare Executive Summary
5. **STOP** -- submit to Owner for approval

### spec.md Template

```markdown
# [Project Name] Specification

## 1. Overview
## 2. Background & Motivation
## 3. Goals
- MUST: [required]
- SHOULD: [expected]
- MAY: [optional]

## 4. Non-Goals
## 5. User Scenarios
## 6. Technical Constraints
## 7. Interface Definitions

### ROS2 Interfaces
#### Subscribed Topics
| Topic | Msg Type | QoS | Frequency |
#### Published Topics
| Topic | Msg Type | QoS | Frequency |
#### Services / Actions / TF Frames / Parameters

## 8. Test Contracts
### Unit Test Contracts
- [ ] `test_<function>`: <what it verifies>
### Integration Test Contracts
- [ ] `test_<node>_<interaction>`: <what it verifies>

## 9. Acceptance Criteria
## 10. Open Questions
```

## Phase 2: Technical Plan (`/sdd plan`)

**Agent**: Architect (Opus)
**Prereq**: spec.md approved
**Output**: `.sdd/plan.md`

```markdown
# [Project Name] Technical Plan

## 1. Architecture Overview
## 2. Technical Decisions
| Component | Choice | Rationale |
## 3. Module Design
## 4. Data Flow
## 5. Directory Structure
## 6. Key Implementation Details
## 7. Test Strategy
- Unit: pure logic, data transforms, message serialization
- Node: launch_testing, single node pub/sub
- Integration: multi-node, end-to-end data flow
## 8. Risks & Mitigations
```

## Phase 3: Task Decomposition (`/sdd tasks`)

**Agent**: Engineer (Sonnet) or Dispatcher
**Prereq**: plan.md approved
**Output**: `.sdd/task.md`

Each task MUST include TDD structure:

```markdown
### Task 1: [Title]
- **Status**: [ ] pending / [~] in progress / [x] completed
- **Agent**: alpha
- **Depends**: none / Task X
- **TDD Deliverables**:
  - RED: `tests/unit/test_foo.py` -- write failing tests
  - GREEN: `src/foo.py` -- minimal implementation to pass
  - REFACTOR: clean up, ensure style compliance
- **Verify**: `colcon build && colcon test --packages-select <pkg>`

## Execution Waves
| Wave | Tasks | Agents | Gate |
|------|-------|--------|------|
| 1 | T1, T2, T4 | Alpha, Beta, Gamma | unit tests pass |
| 2 | T3 | Alpha | unit + integration pass |
```

---

## Phase 4: TDD Execution (`/sdd execute`)

**Agents**: Alpha/Beta/Gamma in parallel

### Per-Task TDD Cycle

```
1. RED:      Write test from spec interface definitions + acceptance criteria
             Run test -> confirm FAIL
2. GREEN:    Write minimal implementation to make tests pass
             Run test -> confirm PASS
3. REFACTOR: Clean code, ensure tests still pass
4. VERIFY:   colcon build && colcon test
5. UPDATE:   Mark task [x], report to Scribe
```

### Wave Execution

```
Wave 1: Launch Alpha, Beta, Gamma in parallel (independent tasks)
  | all unit tests pass
Wave 2: Launch agents for dependent tasks
  | all unit tests pass
Wave N: ...
  | integration tests pass
Review: code-reviewer + security-reviewer in parallel
  | approved
Done -> Completion report to Owner
```

### Parallel Agent Launch

```
Launch simultaneously:
  Agent(Alpha,  "Execute Task T1. TDD cycle.")
  Agent(Beta,   "Execute Task T2. TDD cycle.")
  Agent(Gamma,  "Execute Task T3. TDD cycle.")
  Agent(Scribe, "Track progress for Wave N.")
Wait for all -> verify -> next wave
```

### Escalation

Agent stuck after 2 attempts -> escalate to Architect (Opus).

---

## Phase 5: Verify & Deliver

1. Full build + full test suite
2. Coverage: verify 80%+ (100% for safety-critical)
3. Parallel review: code-reviewer + security-reviewer
4. Scribe: update docs
5. Completion Report to Owner
6. **STOP** -- Owner approves release

### Completion Report

```markdown
## SDD+TDD Complete: [Feature Name]
### One-liner
### Tasks: X/X done
### Test Results
- Unit: X passed / Integration: X passed / Coverage: X%
### Files Changed
### Risks / Known Issues
### Ready for Release: YES / NO
```

## Commands

| Command | Action |
|---------|--------|
| `/sdd init <desc>` | Create `.sdd/`, start spec with test contracts |
| `/sdd spec` | Generate/revise spec |
| `/sdd plan` | Plan with test strategy |
| `/sdd tasks` | Decompose with TDD structure per task |
| `/sdd execute [wave]` | Parallel TDD execution |
| `/sdd status` | Phase + progress + test counts |
| `/sdd review` | Submit for Owner review |
| `/sdd approve` | Approve current phase, advance |
| `/sdd reject <feedback>` | Reject, revise current phase |

## Execution Rules

1. **Spec first**: No code without approved spec.
2. **Review gates**: Each phase reviewed. Agents cannot self-approve.
3. **Docs as truth**: When code contradicts spec.md, spec.md wins.
4. **Change propagation**: spec change -> plan -> tasks -> code.
5. **Task atomicity**: One task, one deliverable.
6. **Parallel by default**: Independent tasks always go to separate agents.
7. **Test first**: Engineer MUST write tests before implementation.
8. **Escalate, don't spin**: Stuck after 2 attempts -> escalate to Architect.

## Scaling

- **Small** (bug fix): spec + plan in one doc, 1 agent, 1 gate
- **Medium** (new node): full spec + plan, 2 agents, unit + integration
- **Large** (new package): full 3-doc pipeline, 3+ agents, all test layers
